## 一、整体开发策略

### 开发原则

- **纵向切片优先**：每个阶段交付一个可运行的垂直功能，而不是先把所有模型写完再写 API。每完成一个阶段都应该能跑起来看到效果。
- **后端先行**：每个功能模块先写后端 API 并用工具（Postman / curl）验证，再写前端页面对接。不要前后端同时开发。
- **数据库优先于业务逻辑**：每次新功能前先确认表结构和 Alembic 迁移文件，表结构稳了再写业务代码。
- **每个阶段结束后打一个 Git Tag**，方便回滚和答辩演示。

### Commit 消息规范

所有 commit 消息统一用以下格式（中文即可）：

```
<类型>(<范围>): <简短描述>

类型：
  feat     新增功能
  fix      修复 bug
  refactor 重构（功能不变）
  test     添加或修改测试
  docs     文档变更
  chore    工程配置（依赖、环境、脚本等）
  db       数据库相关（模型、迁移）
```

示例：
```
feat(tcp): 实现 recv_exactly 粘包安全读取
feat(api): 完成设备 CRUD 接口
fix(ws): 修复 4401 关闭码未触发 Token 刷新的问题
db(migration): 新增 passed_in_count / passed_out_count 字段
```

### Phase 0：工程初始化

**目标**：项目能跑起来，环境确定，目录结构按照设计稿搭好。

#### 步骤

**0.1 创建项目根目录结构**
```
chelljc/
├── backend/
├── frontend/
├── firmware/          # STM32 硬件代码（C/HAL，Keil 或 STM32CubeIDE 工程）
├── docs/
└── .gitignore
```

**0.2 后端环境**
1. 创建 Python 虚拟环境：`python -m venv venv`
2. 安装核心依赖：
   ```
   fastapi uvicorn[standard] sqlalchemy[asyncio] aiomysql alembic
   python-jose[cryptography] passlib[bcrypt] python-dotenv
   ultralytics opencv-python-headless
   ```
3. 生成 `requirements.txt` 和 `requirements-dev.txt`（含 pytest 等）
4. 创建 `backend/.env.example`，按设计稿第13节填写所有变量名（值留空）

**0.3 前端环境**
1. `pnpm create vite@latest frontend --template react-ts`
2. 安装核心依赖：
   ```
   pnpm add tailwindcss @tailwindcss/vite zustand @tanstack/react-query
   react-router-dom axios recharts framer-motion lucide-react
   @fontsource/geist @fontsource/jetbrains-mono @fontsource/orbitron
   ```
3. 安装 shadcn/ui：`pnpm dlx shadcn@latest init`
4. 配置 Tailwind，在 `tailwind.config.ts` 中扩展 HUD 主题色变量
5. 创建 `frontend/.env.example`

**0.4 初始化 Git**
```
git init
git add .
git commit -m "chore: 项目初始化，建立前后端目录结构和依赖配置"
```

### Phase 3：TCP接收 + 推理Pipeline

**目标**：服务端能接收 STM32 的 TCP 帧，完成 YOLO 推理，结果进入队列。这是整个系统最核心的部分。

#### 步骤

**3.1 设备管理路由**（先写，TCP依赖）
- `routers/devices.py`：设备 CRUD + 设备配置读写
- `schemas/device.py`：DeviceCreate、DeviceDetail、DeviceSettingsUpdate

```
git commit -m "feat(api): 完成设备CRUD和设备配置接口"
```

**3.2 推理引擎封装**
- `inference/engine.py`：YOLOv8n + ByteTrack 全局单例，`detect(frame: np.ndarray) -> List[DetectionResult]`
- `DetectionResult` dataclass：tracking_id、class_id、class_name、confidence、bbox
- 线程安全：推理方法加锁或通过 ThreadPoolExecutor 串行调用

```
git commit -m "feat(inference): YOLO+ByteTrack推理引擎封装，全局单例"
```

**3.3 Pipeline 数据结构**
- `pipeline/context.py`：`DevicePipelineContext` dataclass（含 parked_tracker）、`DeviceStats` dataclass

**3.4 TCP 协议解析**
- `tcp/protocol.py`：`recv_exactly()`、帧类型判断（普通帧 / 心跳 / 版本上报）
- `tcp/server.py`：asyncio TCP Server，设备识别（IP查表）、帧分发到 raw_queue、连接断开处理

```
git commit -m "feat(tcp): asyncio TCP Server，粘包安全帧接收"
git commit -m "feat(tcp): 心跳包ACK、版本上报处理、未注册IP拒绝"
```

**3.5 推理协程**
- `pipeline/inference_loop.py`：从 raw_queue 取帧 → run_in_executor 推理 → 结果写 result_queue

**3.6 分发协程**
- `pipeline/dispatch_loop.py`：从 result_queue 取结果 → 同时写 ws_queue 和 db_queue

**3.7 PipelineManager**
- `pipeline/manager.py`：`on_device_connected()`、`on_device_disconnected()`、`invalidate_settings_cache()`

```
git commit -m "feat(pipeline): 每设备独立Pipeline，推理+分发协程"
git commit -m "feat(pipeline): PipelineManager设备上下线管理"
```

**3.8 整合到 `main.py`**
- FastAPI lifespan 中启动 TCP Server 和 PipelineManager

**验证**：用 Python 脚本模拟 STM32（发送图片帧），在日志中确认帧被接收并完成推理，result_queue 有数据。

### Phase 4：核心业务逻辑

**目标**：虚拟线计数、异常停车检测、全套预警机制完整可用。

#### 步骤

**4.1 虚拟线计数**
- `inference/counter.py`：纯函数 `count_crossings(vehicles, line_y, prev_centers) -> (in_count, out_count, updated_centers)`
- 逻辑：比较当前帧和上一帧的bbox中心点Y坐标与line_y的相对位置变化

**4.2 异常停车检测**
- `inference/parking_detector.py`：`ParkingDetector` 类
- 方法：`check(tracking_id, center, timestamp, park_timeout) -> bool`
- 内部维护静止时长字典（使用 `DevicePipelineContext.parked_tracker`，由外部传入，不持有状态）

```
git commit -m "feat(inference): 虚拟线双向计数（passed_in/out）"
git commit -m "feat(inference): 异常停车检测逻辑"
```

**4.3 预警触发 + 解除**
- `services/alert_resolver.py`：`AlertResolver` 全局单例
- 拥堵预警：L2/L3/L4 阈值判断 + 10秒持续确认 + 30秒迟滞解除
- 异常停车预警：tracking_id消失或恢复移动自动解除
- flow_spike：历史均值计算（含冷启动保护，历史数据 < 7天跳过）
- flow_zero：5分钟无检测触发
- device_offline：TCP断线时触发

```
git commit -m "feat(alert): 拥堵预警触发和迟滞解除机制"
git commit -m "feat(alert): 异常停车/flow_spike/flow_zero/离线预警"
```

**4.4 将计数和预警接入 dispatch_loop**
- `pipeline/dispatch_loop.py` 调用 counter、parking_detector、alert_resolver

**验证**：模拟帧数据，用断点/日志确认：过线计数正确区分方向 → 静止超30秒触发停车预警 → 阈值以下30秒后拥堵自动解除。

**5.3 数据清理任务**
- `services/cleanup.py`：每日凌晨3:00批量清理（每批1000行）

### Phase 6：WebSocket推送

**目标**：前端能通过 WebSocket 实时收到帧数据、预警通知、健康数据。

#### 步骤

**6.1 WebSocket 连接管理**
- `services/websocket_manager.py`：按 device_id 分组存储连接，支持 broadcast(device_id, msg) 和 broadcast_all(msg)

**6.2 WebSocket 推送协程**
- `pipeline/ws_push_loop.py`：从 ws_queue 取结果，base64编码帧，打包成 `stream_frame` JSON，通过 WebSocketManager 广播

**6.3 WebSocket 路由**
- `/ws/stream/{device_id}`：鉴权（URL token参数）、握手、订阅、Token即将过期推送（剩余5分钟时）、ping/pong心跳
- `/ws/health`：每秒推送 `health_report`（CPU/GPU/各设备队列状态/降级等级）

```
git commit -m "feat(ws): WebSocketManager按设备分组广播"

git commit -m "feat(ws): /ws/health 健康数据每秒推送"
```

**6.4 性能降级监控**

git commit -m "feat(pipeline): 三级性能降级机制和状态监控"
