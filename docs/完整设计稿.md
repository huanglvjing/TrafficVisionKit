## 一、系统概述

本系统是一个面向交通监控场景的端到端智能视频分析平台，实现从嵌入式设备图像采集、AI推理、实时可视化到历史数据查询的完整闭环。系统采用高内聚低耦合架构，强调实时性、稳定性与可维护性。

- **硬件层**：STM32F407 + OV2640 摄像头 + W5500 以太网模块，固定 IP `192.168.1.100`，通过 TCP 发送 JPEG 帧至服务端。
- **服务层**：部署于 `192.168.1.10`，基于 FastAPI 构建异步服务，集成 YOLOv8n + ByteTrack 实现车辆检测与跟踪，支持虚拟线计数与异常停车预警。
- **展示层**：React 18 + TypeScript 前端，通过 WebSocket 接收实时视频流与统计指标，支持 Recharts 图表、Zustand 状态管理、TanStack Query 数据请求。
- **扩展能力**：支持多设备接入（每设备独立队列集与协程组）、运行时配置热更新、两级数据聚合、细粒度预警机制、设备在线率追踪，并具备完整的用户认证与权限管理体系。

## 三、完整数据流

### 3.1 STM32 采集并发送

OV2640 DCMI → STM32 硬件 JPEG 压缩 → 加 4 字节大端包头（uint32 帧字节数，`0xFFFFFFFF` 为心跳包）→ W5500 TCP 发往 `192.168.1.10:9000`，目标 25~30fps。

每帧打包格式：

- 字节 0~3：帧长度（uint32，大端序）。值为 `0xFFFFFFFF` 时为心跳包，无 payload；值为 `0xFFFFFFFE` 时为固件版本上报包，payload 为 ASCII 版本字符串（最长 20 字节）。
- 字节 4~N：JPEG 帧原始数据（N = 帧长度）。

### 3.2 服务器 TCP 接收（粘包安全）

FastAPI 启动时同步启动 asyncio TCP Server 监听 9000 端口。`recv_exactly(reader, n)` 循环读取直到够 n 字节，彻底避免粘包。处理流程：

1. 先读 4 字节包头解析长度。
2. 若为心跳包（`0xFFFFFFFF`），回应 4 字节 ACK（`0x00000001`）并重置超时计时器（60s 无数据则主动断开）。
3. 若为版本上报包（`0xFFFFFFFE`），读取 payload，更新 `devices.firmware_version` 字段。
4. 否则按长度精确读取完整 JPEG 帧，通过远端 IP 查找对应 `DevicePipelineContext`，将帧推入该设备专属的 `raw_queue`（maxsize=2）；队列满则丢弃最旧帧并累加 `dropped_frame` 计数器。
5. 连接断开时写 `connection_sessions` 记录，更新 `devices.is_active=0`，通过 WebSocket 向前端推送断线通知。

TCP Server 通过连接的远端 IP 识别设备（`reader.transport.get_extra_info("peername")[0]`），再查 `devices` 表获取 `device_id`。若 IP 未注册，记录警告日志并立即断开。

### 3.3 YOLO 推理（不阻塞事件循环）

每个设备的推理协程从自身 `raw_queue` 取帧，通过 `loop.run_in_executor(shared_thread_pool)` 提交 YOLOv8n + ByteTrack 推理（GPU 约 12ms/帧），推理完毕后在协程中执行虚拟线计数、异常停车检测、OpenCV 绘帧，JPEG 编码后将结果推入 `result_queue`。

- **虚拟线计数**：读取 `device_settings.line_y`（运行时从内存热缓存中获取，设置页保存后立即失效并重载），判断每个 `tracking_id` 的 bbox 中心点是否跨越该 Y 坐标，记录跨越方向：bbox 中心点 Y 坐标从小于 `line_y` 变为大于等于 `line_y` 为**向下穿越**（进入方向，计入 `passed_in_count`）；反之为**向上穿越**（驶出方向，计入 `passed_out_count`）。每帧结果均写入内存聚合缓冲区，60 秒聚合时三个字段同步写入 `traffic_records`（`passed_count = passed_in_count + passed_out_count`）。
- **异常停车检测**：维护 per-device 内存字典 `parked_tracker[tracking_id] = {last_center, still_since}`，每帧计算位移；位移小于阈值（默认 5px）则累计静止时长；超过 `park_timeout_seconds`（默认 30s，从热缓存读取）且该车辆尚无未解除预警时，写入 `traffic_alerts` 并推送 L3 级预警。
- **推理线程池**：共享一个 `ThreadPoolExecutor(max_workers=2)`（单 GPU 场景避免过多并发争抢显存），多设备推理协程均使用此线程池，GPU 串行执行。

### 3.4 分发结果（三路独立）

分发协程从 `result_queue` 取出帧结果，同时写入 `ws_queue`（推前端画面）和 `db_queue`（持久化数据）。三路消费者完全独立，DB 写慢不影响视频流推送。每路队列均设 maxsize=2，满时丢帧并累加计数器。

### 3.5 WebSocket 推送前端

`ws_queue` 消费协程将检测帧 base64 编码，与当前统计数据打包成 JSON，通过 `WebSocketManager` 广播给订阅该 `device_id` 的所有前端连接。每秒额外通过 `ws/health` 推送一次系统健康数据。WebSocket 消息格式详见第五节。

### 3.6 写入 MySQL（两级聚合）

`db_queue` 消费协程将每帧的检测结果写入内存聚合缓冲区。定时任务：

- 每 60 秒聚合缓冲区数据 → 写 `traffic_records` 一条（avg_count / max_count / passed_count / 车型细分）。
- 每 3600 秒从 `traffic_records` 聚合 → 写 `hourly_statistics` 一条。
- 预警触发/解除时实时写入 `traffic_alerts`。
- 历史查询直接走聚合表，毫秒响应。

### 3.7 前端历史查询

TanStack Query 发 GET `/api/history/traffic`，带 `start` / `end` / `granularity` 参数，FastAPI 按粒度查对应聚合表返回 JSON，Recharts 渲染折线图 / 柱状图 / 热力时间图。

## 六、JWT 双 Token 机制【新增补充】

> 原规范中仅有 1 小时 access_token，无刷新方案，会导致用户操作中途被强制登出。本节补充完整的双 Token 方案。
> 

### 6.1 Token 设计

| 类型 | 有效期 | 存储位置 | 用途 |
| --- | --- | --- | --- |
| access_token | 60 分钟 | 前端内存（Zustand useAuthStore） | 携带在每条 REST 请求的 `Authorization: Bearer <token>` Header 中，以及 WebSocket 连接的 URL query 参数中 |
| refresh_token | 7 天 | HttpOnly Cookie（Secure，SameSite=Strict） | 仅用于调用 `/api/auth/refresh` 换取新 access_token，不携带到其他接口 |
- **access_token 存内存**：防止 XSS 攻击窃取（localStorage 有 XSS 风险）。页面刷新后 access_token 丢失，App 初始化时自动调用 `/api/auth/refresh` 静默续期。
- **refresh_token 存 HttpOnly Cookie**：JavaScript 代码无法读取，防止脚本泄漏。浏览器自动携带，无需手动处理。

### 6.2 Token 刷新流程（REST 接口）

1. 前端发起 REST 请求，服务端返回 401（access_token 过期）。
2. 前端 axios 拦截器捕获 401，暂停当前请求，自动调用 `POST /api/auth/refresh`（浏览器自动携带 Cookie 中的 refresh_token）。
3. 服务端验证 refresh_token：未过期且不在内存黑名单中 → 签发新的 access_token（60 分钟）和新的 refresh_token（7 天，旋转策略）→ 返回 200。
4. 前端用新 access_token 更新 store，并重试原请求。整个过程对用户无感知。
5. 若 refresh_token 也过期或无效 → 服务端返回 401 业务码 1005 → 前端清除认证状态，跳转登录页。

### 6.3 Token 旋转策略（Refresh Token Rotation）

每次调用 `/api/auth/refresh` 后，旧的 refresh_token 立即加入服务端内存黑名单（TTL = 7 天后自动清除）。若检测到一个已被使用过的旧 refresh_token 再次被提交（说明 Token 可能已被泄漏），服务端立即吊销该用户所有 Token，强制用户重新登录，并写入 `user_failed_login` 类型的 system_logs。

> **⚠ 已知限制**：refresh_token 黑名单存储在进程内存中，服务重启后黑名单清空，已旋转吊销的旧 refresh_token 将重新可用（有效期内）。本系统为单机内网部署，此限制在可接受范围内。若升级为生产多实例部署，需将黑名单迁移至 Redis 或数据库持久化存储。

### 6.4 WebSocket Token 刷新

WebSocket 长连接持续期间 access_token 可能到期。处理策略：

1. 服务端在 Token 有效期剩余 5 分钟时，通过 WebSocket 下发 `token_expiring` 消息（见 5.2 节）。
2. 前端 `useWebSocket` Hook 收到通知后，在后台静默调用 `/api/auth/refresh` 拿到新 Token。
3. 前端主动断开并用新 Token 重新连接 WebSocket（用户无感知，断连时间 < 1 秒）。
4. 若服务端直接收到 Token 已过期的连接请求，拒绝握手并发送 close code 4401，前端重连逻辑自动触发 Token 刷新。

## 八、数据保留与清理策略【新增补充】

> 原规范仅对 system_logs 说明了 30 天保留策略，traffic_records 等高频写入表无清理策略，长期运行将大量消耗磁盘空间。
> 

### 8.1 各表保留策略

| 表名 | 写入频率 | 保留周期 | 预估年数据量（单设备） | 清理方式 |
| --- | --- | --- | --- | --- |
| traffic_records | 1条/分钟 | 最近 90 天 | 约 43,800 行 | 定时任务自动删除 |
| hourly_statistics | 1条/小时 | 永久保留 | 约 8,760 行 | 不清理 |
| traffic_alerts | 触发即写 | 最近 180 天 | 视预警频率而定 | 定时任务自动删除 |
| connection_sessions | 每次连接 | 最近 90 天 | 视连接次数而定 | 定时任务自动删除 |
| system_logs | 实时写入 | 最近 30 天 | 视操作频率而定 | 定时任务自动删除 |

### 8.2 定时清理任务实现

清理任务作为 asyncio 定时协程，在 FastAPI 启动时通过 `asyncio.create_task()` 启动，每日凌晨 3:00 执行一次。

实现思路：计算距离下一个凌晨 3:00 的秒数，`await asyncio.sleep()` 等待，时间到后依次对 `traffic_records`（90天）、`traffic_alerts`（180天）、`connection_sessions`（90天）、`system_logs`（30天）执行清理。

每次清理使用批量删除方式（每批 1000 行，循环执行直到无剩余），避免单次大量 DELETE 造成表锁，影响正常写入。具体语句形式为 `DELETE FROM {table} WHERE created_at < :cutoff LIMIT 1000`，循环至受影响行数为 0 为止。

## 九、预警自动解除逻辑【新增补充】

> 原规范 traffic_alerts 表有 resolved_at 和 is_resolved 字段，但未说明解除触发条件和实现机制。
> 

### 9.1 各类预警的解除条件

| alert_type | 触发条件 | 自动解除条件 | 解除方式标记 |
| --- | --- | --- | --- |
| congestion（拥堵） | 车辆数 ≥ L2/L3/L4 阈值，持续 10 秒 | 连续 30 秒车辆数低于（触发阈值 - 1），迟滞解除，防止频繁抖动 | auto |
| abnormal_stop（异常停车） | 某 tracking_id 静止超过 park_timeout_seconds | 该 tracking_id 从画面消失（ByteTrack 不再追踪），或恢复移动（位移 > 20px）持续 5 秒 | auto |
| flow_spike（流量突增） | 1分钟内过线车辆数 > 历史同期均值 300%（历史均值计算见下方说明） | 下一分钟流量恢复正常范围（< 150% 历史均值） | auto |
| flow_zero（零流量） | 设备在线但连续 5 分钟无车辆检测 | 检测到至少 1 辆车辆 | auto |
| device_offline（设备离线） | TCP 连接断开 | 设备重新建立 TCP 连接 | auto |
| 所有类型 | 任意触发状态 | 操作员在前端手动点击「标记已处理」 | manual |

### 9.2 flow_spike 历史均值计算说明

**历史均值定义**：取 `hourly_statistics` 表中，同一设备、同一小时时段（hour_of_day = 当前小时数，如14时对应所有历史14:00条目）、最近 4 周内的 `total_passed / 60`（折算为每分钟均值）的平均值。

**冷启动保护**：若可用的历史同期数据条数 < 7（即历史数据不足一周），则跳过 flow_spike 检测，不触发预警。待历史数据积累满 7 天后自动启用。`aggregator.py` 在每分钟聚合写入后检查此条件，不足时在 `system_logs` 中写入一条 `event_type=info`、`message="flow_spike 检测冷启动中，历史数据不足"` 的记录（每小时仅写一次，避免日志刷屏）。

### 9.3 迟滞解除机制说明

拥堵预警使用迟滞（Hysteresis）解除而非即时解除，原因：车辆数在阈值附近小幅波动时（如阈值10辆，检测结果在9~11辆间来回），若即时解除会导致预警频繁触发和解除，产生大量噪声记录。迟滞解除要求：车辆数必须连续低于阈值 30 秒后，才判定为真实解除。

### 9.4 实现架构

在 `services/alert_resolver.py` 中实现 `AlertResolver` 类（全局单例），内部维护：

- `active_alerts: dict[device_id, dict[alert_type, alert_id]]`：记录各设备当前未解除的预警。
- `hysteresis_counter: dict[alert_id, float]`：拥堵预警的迟滞计数器，记录持续低于阈值的秒数。

每帧推理结果经 dispatch_loop 处理后，调用 `alert_resolver.check_and_resolve(device_id, detection_result)`，检查所有活跃预警的解除条件，满足时调用 `resolve_alert(alert_id, "auto")`：更新数据库记录（写入 resolved_at、duration_seconds、resolved_by）→ 通过 WebSocketManager 广播 `alert_resolved` 消息 → 从内存活跃预警表中移除。

同一设备同一 alert_type 在有未解除的预警存在时，不重复创建新预警记录（防重复），仅在解除后才允许再次触发。

## 十、多设备并发架构【新增补充】

> 原规范多处提到"支持多设备接入"，但队列是全局共享还是每设备独立未明确，TCP 如何区分设备也未说明。
> 

### 10.1 核心设计：每设备独立 Pipeline

每个 STM32 设备建立 TCP 连接后，服务端通过 `PipelineManager` 为其创建一个 `DevicePipelineContext`，包含该设备专属的四条 asyncio 队列和四个异步任务，彼此完全隔离，一个设备卡顿不影响其他设备。

**DevicePipelineContext 结构：**

`@dataclass
class DevicePipelineContext:
    device_id: int
    device_ip: str
    raw_queue: asyncio.Queue      # maxsize=2，存放原始 JPEG 帧
    result_queue: asyncio.Queue   # maxsize=2，存放推理结果（含检测框渲染帧）
    ws_queue: asyncio.Queue       # maxsize=2，存放 WebSocket 推送数据
    db_queue: asyncio.Queue       # maxsize=2，存放数据库写入任务
    inference_task: Optional[asyncio.Task] = field(default=None)  # 该设备的推理协程 Task
    dispatch_task: Optional[asyncio.Task]  = field(default=None)  # 该设备的分发协程 Task
    ws_task: Optional[asyncio.Task]        = field(default=None)  # 该设备的 WebSocket 推送协程 Task
    db_task: Optional[asyncio.Task]        = field(default=None)  # 该设备的 DB 写入协程 Task
    settings_cache: dict = field(default_factory=dict)  # 热缓存的设备配置，PUT settings 后强制失效重载
    stats: DeviceStats   = field(default_factory=DeviceStats)  # 运行时统计（帧率、丢帧数、推理延迟等）
    parked_tracker: dict = field(default_factory=dict)
    # 异常停车检测状态，格式：{tracking_id: {"last_center": (x, y), "still_since": float}}
    # 此字典与 DevicePipelineContext 同生命周期：设备断线时随 Context 一并销毁，
    # 重连后全部清零，避免旧 tracking_id 与新 ByteTrack ID 碰撞导致误触发停车预警。`

**PipelineManager 职责：**

- `on_device_connected(device_id, device_ip)`：创建 `DevicePipelineContext`，启动四个异步 Task，注册到全局字典。
- `on_device_disconnected(device_id)`：取消并等待四个 Task 结束（`asyncio.gather(..., return_exceptions=True)`），从全局字典移除。
- `invalidate_settings_cache(device_id)`：清除指定设备的配置热缓存，下一帧自动从数据库重载。

### 10.2 共享资源列表

| 资源 | 共享方式 | 说明 |
| --- | --- | --- |
| YOLO 模型实例 | 全局单例 | 模型加载一次；GPU 推理内部串行（单卡），通过线程池排队 |
| ThreadPoolExecutor | 全局共享，max_workers=2 | 避免多设备并发推理同时占用 GPU 造成显存溢出 |
| WebSocketManager | 全局共享 | 维护所有前端 WebSocket 连接，内部按 device_id 分组存储，支持按设备广播 |
| 数据库连接池 | 全局共享 | aiomysql 连接池（minsize=3, maxsize=10），各设备 db_write_loop 竞争获取连接 |
| AlertResolver | 全局共享 | 内部按 device_id 隔离预警状态数据 |

### 10.3 TCP 多设备识别机制

TCP Server 通过 `reader.transport.get_extra_info("peername")[0]` 获取远端 IP，查询 `devices` 表的 `ip_address` 字段找到对应 `device_id`。若远端 IP 未注册，写入 `system_logs`（event_type=warning，message="未知设备尝试连接"），并立即断开连接，不进入推理 Pipeline。

## 十一、性能降级策略【新增补充】

> 原规范仅提到 raw_queue 满时丢帧，缺乏系统性的降级与限流方案。
> 

### 11.1 三级降级机制

| 级别 | 触发条件 | 采取措施 | 恢复条件 |
| --- | --- | --- | --- |
| 正常 | 所有队列深度 0~1，推理延迟 < 20ms | 正常处理所有帧 | — |
| L1 降级（轻度） | 任意队列深度连续 3 秒 ≥ 2，或推理延迟 > 30ms | 限制入队频率，跳过每隔一帧（即实际处理帧率降为原来的 50%） | 队列深度连续 10 秒 < 1 |
| L2 降级（严重） | 推理延迟连续 > 50ms，或 GPU 利用率 > 95% | 暂停 db_write_loop（db_queue 不再被消费），优先保证视频推送；db_queue 满时丢弃最旧任务 | 推理延迟 < 30ms 且 GPU < 80% |
| L3 降级（紧急） | 系统内存 > 90% 或 GPU 显存 OOM 异常 | 停止所有设备的推理协程，向前端推送 system_error 消息，等待管理员介入重启 | 需手动重启服务 |

降级状态通过健康数据消息（health_report）中的 `degradation_level` 字段（0~3）实时告知前端。

### 11.2 fps_limit 令牌桶限流

`device_settings.fps_limit`（默认 30fps）控制服务端对该设备的处理帧率上限。TCP 接收协程通过令牌桶算法（Token Bucket）限制入队速率：

- 令牌桶容量 = fps_limit，以 fps_limit 个/秒的速率补充令牌。
- 每帧到达时消耗 1 个令牌；令牌不足则直接丢弃该帧（不入队），不消耗队列容量。
- fps_limit 从设备配置热缓存读取，PUT settings 后立即生效，无需重启。

### 11.3 前端降级感知

- 健康数据消息中的 `dropped_frames > 10`（每分钟）时，视频面板顶部显示「帧率降级」黄色标签。
- `avg_inference_ms > 40` 时，健康状态面板对应指标显示橙色警告色。
- 服务端降级至 L3 时，前端展示全屏遮罩提示「推理服务异常，请联系管理员」，并停止重连 WebSocket。

| 模块 | 型号/参数 | 说明 |
| --- | --- | --- |
| 主控 | STM32F407VGT6 | 主频 168MHz，带 FPU |
| 摄像头 | OV2640 | DCMI 接口，JPEG 硬件压缩输出，目标分辨率 640×480 |

### 12.2 TCP 数据帧格式（完整）

| 包类型 | frame_length 值 | payload | 说明 |
| --- | --- | --- | --- |
| 普通图像帧 | 实际 JPEG 字节数（uint32 大端序） | JPEG 数据 | 正常采集帧 |
| 心跳包 | 0xFFFFFFFF | 无 | 通知服务端设备存活 |
| 版本上报包 | 0xFFFFFFFE | ASCII 版本字符串，最长 20 字节，如 "v1.2.3" | 每次连接建立后、第一帧前发送一次 |

所有包均以 4 字节大端序 `frame_length` 开头，后跟 `frame_length` 字节的 payload。

### 12.3 固件发送流程

1. TCP 连接建立后，立即发送版本上报包（`0xFFFFFFFE` + 版本字符串）。
2. 发送第一帧心跳包（`0xFFFFFFFF`，0 字节 payload），通知服务端设备就绪。
3. 正常工作时，每帧图像采集完成后：计算 JPEG 长度 → 写 4 字节大端序包头 → 拼接 JPEG 数据 → 调用 W5500 TCP Send。
4. 若当前无图像帧（如帧率控制等待期），每 30 秒至少发送一次心跳包，防止服务端 60 秒超时断开。
5. 收到服务端 ACK（4 字节 `0x00000001`）视为心跳应答正常。
6. TCP 连接异常断开时，等待 2 秒后重新发起连接（前 10 次每次间隔 2 秒，之后改为每 30 秒重试一次，无限重试直到成功）。

**规范一：配置集中管理**

**规范二：异常统一处理**
在 `core/exceptions.py` 中定义 `AppException(code: int, message: str, status_code: int)` 基类，所有业务异常继承此类（如 `DeviceNotFoundException`、`DuplicateIPException`）。在 `main.py` 中注册全局异常处理器，所有异常均以统一格式响应：`{"code": int, "message": str, "detail": str?}`，不在路由中写 `try/except` 处理业务异常。

**规范三：日志统一入口**`core/logger.py` 封装 `get_logger(name: str)` 函数，返回的 Logger 实例同时输出到标准输出（供 uvicorn 日志系统采集）和异步写入 `system_logs` 表。写入数据库时携带上下文：`event_type`、`device_id`（如有）、`operator_ip`（如有）、`message`。业务代码中**禁止**直接使用 `print()` 或 `logging.getLogger()`。

**规范四：路由瘦化原则**
路由函数（`routers/*.py`）只做三件事：接收 Request → 调用 Service 层函数 → 返回 Response Schema。所有业务逻辑、数据查询、计算均封装在 `services/` 或 `inference/` 中。单个路由函数代码行数不超过 30 行，超出则提取到 Service 层。

**规范五：推理模块完全解耦**`inference/` 目录中所有模块不依赖 FastAPI、SQLAlchemy 或任何 Web/DB 框架组件。`engine.py` 只接受 `np.ndarray` 输入，返回纯 Python dataclass 列表（`List[DetectionResult]`）。`counter.py` 和 `parking_detector.py` 均为纯业务逻辑，不涉及 I/O。未来若替换为 ONNX / TensorRT 推理引擎，只需修改 `engine.py`，其余代码零改动。

**规范六：设备配置热缓存**`DevicePipelineContext.settings_cache` 字典在推理协程首次运行时从数据库加载，此后每帧直接读取内存缓存。`PUT /api/devices/{device_id}/settings` 接口在写入数据库后，必须调用 `PipelineManager.invalidate_settings_cache(device_id)`，使缓存失效，下一帧的推理协程在发现缓存为空时自动重载。

**规范七：Pydantic Schema 严格分层**
数据库 ORM 模型（`models/`）与 API 数据模型（`schemas/`）严格分离，路由函数中不直接序列化 ORM 对象。`schemas/common.py` 提供泛型分页响应 `PagedResponse[T]`，所有分页接口统一响应格式：`{"total": int, "page": int, "size": int, "items": [...]}`.

**规范八：禁止全局可变状态**
除 `PipelineManager`、`WebSocketManager`、`AlertResolver` 这三个有明确生命周期管理的单例外，禁止模块级别的全局可变变量。各 Pipeline 的运行时状态（帧率、丢帧数等）均存放在 `DevicePipelineContext.stats` 中，通过 `PipelineManager` 统一访问。

### 15.4 前端关键开发规范

**规范一：TypeScript 严格模式**`tsconfig.json` 开启 `strict: true`，禁止使用 `any`（必须使用时需加 `// eslint-disable-next-line @typescript-eslint/no-explicit-any` 注释并说明原因）。所有组件 Props 定义独立 `interface`，禁止 inline object type。

**规范二：组件纯粹化与 Props 驱动**
所有 UI 组件只接受 props，不在组件内部直接调用 store（容器组件和页面组件除外）或发起 API 请求。`StatCard` 通过 `format` 函数 prop 自定义显示格式，通过 `alertLevel` prop 映射颜色，不硬编码任何颜色值或格式规则。图表组件只接受数据 prop，对数据来源不作假设。

**规范三：WebSocket Hook 分层**
底层 `useWebSocket` 只负责连接管理（连接、断开、重连、Token 刷新触发）和原始消息回调，不解析业务消息。业务层 `useStreamSocket` 和 `useHealthSocket` 负责消息解析和状态写入，不处理连接细节。页面组件调用业务层 Hook，不直接使用底层 `useWebSocket`。

**规范四：权限控制集中化**
所有权限判断集中在 `routes/index.tsx` 中的 `ProtectedRoute`（未登录跳转）和 `AdminRoute`（非 admin 跳首页并 Toast 提示）。业务组件内**禁止**写 `if (user.role === 'admin')` 的判断逻辑，菜单可见性也通过路由配置控制而非组件内判断。

**规范五：CSS 变量统一样式**
所有主题颜色通过 `main.tsx` 的 `:root` CSS 变量定义（`--bg-base`、`--color-accent`、`--color-danger` 等），Tailwind 的 `extend` 中引用这些变量（如 `accent: 'var(--color-accent)'`）。业务代码**禁止**使用 `style={{ color: '#00D4FF' }}` 形式的内联硬编码颜色。颜色值只存在一个地方：`main.tsx` 的 `:root`。

**规范六：错误边界与防御解析**`App.tsx` 包裹全局 React `ErrorBoundary`，捕获渲染阶段异常，显示友好错误页而非白屏。所有 WebSocket 消息解析外层包裹 `try/catch`，解析失败只打印 `console.warn`，不 throw，防止一条格式错误的消息导致整条 WebSocket 连接中断。

**规范七：TanStack Query 统一管理**
所有 REST 请求通过 `lib/api.ts` 中封装的 Query/Mutation hooks 发起，不在组件或其他 hook 中直接调用 axios。`queryKey` 设计规范：`['devices']`、`['device', id]`、`['device', id, 'settings']`、`['history', 'traffic', params]` 等，便于精准 invalidate 和缓存控制。

## 十七、安全与错误处理

### 17.1 安全加固

- **JWT 双 Token**：access_token 存内存（1小时），refresh_token 存 HttpOnly Cookie（7天），详见第六节。
- **密码安全**：bcrypt（cost=12）哈希存储，密码强度：长度 ≥ 8，含大小写字母及数字。
- **登录防爆破**：连续失败 5 次 → 账号锁定 10 分钟（内存 TTL 缓存，key 为 username）。锁定期间即使密码正确也返回 423 + 错误码 1009。
- **CORS**：所有 REST API 和 WebSocket 启用 CORS，仅允许 `CORS_ORIGINS` 环境变量指定的来源。
- **WebSocket 鉴权**：连接握手时验证 access_token，过期或无效时发送 close code 4401 并拒绝连接。
- **SQL 注入防护**：所有数据库操作通过 SQLAlchemy ORM 或参数化查询，禁止字符串拼接构造 SQL。
- **生产环境**：强制 HTTPS，Cookie 设置 `Secure` 和 `HttpOnly` 标志，`SameSite=Strict`。

### 17.2 API 错误码规范

| HTTP 状态码 | 业务码 | 含义 |
| --- | --- | --- |
| 200 | 0 | 成功 |
| 400 | 1001 | 参数校验失败（请求体格式错误） |
| 401 | 1005 | 无效凭据或 Token 过期 |
| 403 | 1006 | 权限不足（角色不够） |
| 404 | 1002 | 设备不存在 / 资源不存在 |
| 409 | 1003 | IP 地址重复（创建设备时） |
| 409 | 1007 | 用户名已存在（创建用户时） |
| 422 | 1004 | 配置值越界（如 line_y 超出分辨率范围） |
| 422 | 1008 | 密码强度不足 |
| 423 | 1009 | 账号被锁定（登录失败次数超限，附带 locked_until 时间戳） |
| 500 | 5001 | 数据库操作异常 |
| 500 | 5002 | YOLO 推理异常 |

> **统一响应格式**：`{"code": int, "message": str, "detail": str?}`
前端 axios 拦截器统一处理：401 → Token 刷新流程；403 → Toast 提示「权限不足」；423 → Toast 提示「账号锁定，X 分钟后重试」；500 → Toast 提示「服务器异常，请稍后重试」。

## 十八、部署与初始化

### 18.1 首次部署流程

1. 安装 MySQL 8.0，创建数据库：`CREATE DATABASE traffic_detection CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;`
2. 配置 `backend/.env`（参考 `.env.example`，重点填写 `DATABASE_URL` 和 `JWT_SECRET_KEY`）。
3. 安装 Python 依赖：`pip install -r requirements.txt`
4. 执行初始化脚本：`python backend/init_db.py`（内部执行 `alembic upgrade head` + 创建默认 admin 账号 + 写入默认设备记录）。
5. 启动 FastAPI：`uvicorn main:app --host 0.0.0.0 --port 8000`
6. 配置 `frontend/.env`（填写 `VITE_API_BASE_URL` 和 `VITE_WS_BASE_URL`）。
7. 安装前端依赖：`pnpm install`
8. 开发模式：`pnpm dev`；生产模式：`pnpm build`，产物在 `dist/` 目录。

### 18.2 升级部署流程（已有生产数据）

1. 拉取新版本代码。
2. 停止后端服务（`systemctl stop traffic-backend` 或 `pm2 stop traffic-backend`）。
3. 执行数据库迁移：`alembic upgrade head`。若失败立即执行 `alembic downgrade -1` 并回滚代码。
4. 安装新依赖：`pip install -r requirements.txt`。
5. 重启后端服务：`systemctl start traffic-backend`。
6. 前端构建：`pnpm build`，将 `dist/` 目录同步至 Nginx 静态目录。

### 18.3 生产部署建议

| 方面 | 建议方案 | 说明 |
| --- | --- | --- |
| 进程守护 | systemd 或 PM2 | systemd service 文件托管 uvicorn 进程，崩溃自动重启，日志写入 journald |
| 反向代理 | Nginx | 代理 `/api/*` 和 `/ws/*` 至 8000 端口，WebSocket 需配置 `proxy_read_timeout 300s` 和 `proxy_http_version 1.1` + `Upgrade` Header |
| HTTPS | Let's Encrypt + Certbot 或内网自签证书 | 生产环境必须开启，同时将前端 `VITE_WS_BASE_URL` 改为 `wss://`，后端 `COOKIE_SECURE=true` |
| 前端静态 | Nginx 静态文件 + gzip 压缩 | 构建产物放 `/var/www/traffic-frontend/`，配置 `try_files $uri $uri/ /index.html` 支持 SPA 路由 |
| 日志管理 | uvicorn --log-config 按日期轮转 | 保留最近 7 天日志文件，配合 logrotate |
| Nginx 日志脱敏 | `log_format` 过滤 token 参数 | WebSocket 连接 URL 中包含 `?token=...`，Nginx access_log 会明文记录 Token。需在 `nginx.conf` 中自定义 `log_format`，将 `$args` 替换为脱敏版本（将 `token=xxx` 替换为 `token=***`），或使用 `$uri`（不含查询参数）记录日志，防止 Token 出现在日志文件中 |
| 数据库备份 | mysqldump 每日全量备份 | 定时任务凌晨 2:00 执行，备份文件保留 30 天，存储在非系统盘 |
